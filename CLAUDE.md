# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

ArcKit is an **Enterprise Architecture Governance & Vendor Procurement Toolkit** providing slash commands for AI coding assistants (Claude Code, Codex CLI, Gemini CLI, OpenCode CLI, GitHub Copilot, Paperclip AI) to generate architecture artifacts.

**Seven distribution formats** exist side-by-side in this repo:

1. **CLI package** (`src/arckit_cli/`) — Python CLI (`arckit init`) scaffolding projects for Codex/OpenCode/Copilot
2. **Claude Code plugin** (`arckit-claude/`) — installed via `/plugin marketplace add tractorjuice/arc-kit`
3. **Gemini CLI extension** (`arckit-gemini/`) — `gemini extensions install tractorjuice/arckit-gemini`
4. **OpenCode CLI extension** (`arckit-opencode/`) — scaffolded via `arckit init --ai opencode`
5. **Codex CLI extension** (`arckit-codex/`) — published as `tractorjuice/arckit-codex`
6. **Copilot extension** (`arckit-copilot/`) — prompt files + custom agents for GitHub Copilot
7. **Paperclip AI plugin** (`arckit-paperclip/`) — TypeScript plugin built with `npm run build`

The plugin source (`arckit-claude/`) is canonical. Non-Claude formats are generated by `scripts/converter.py`. CLI and plugin have independent version numbers; all extensions track the plugin.

## Build & Development Commands

```bash
# Install CLI in development mode (uv recommended)
uv pip install -e .

# Test CLI
arckit --help
arckit check

# Test project initialization
arckit init test-project --ai codex --no-git
arckit init test-project --ai codex --minimal  # Skip docs/guides

# Build CLI distribution
python -m build

# Build Paperclip plugin (TypeScript)
cd arckit-paperclip && npm run build

# Convert plugin to Codex/OpenCode/Gemini/Copilot formats
python scripts/converter.py

# Lint markdown files
npx markdownlint-cli2 "**/*.md"
npx markdownlint-cli2 --fix "**/*.md"
```

## Architecture

### Core Components

**CLI Entry Point** (`src/arckit_cli/__init__.py`):

- Single-file CLI using `typer`. `arckit init` scaffolds project structure and copies templates/commands from package data. `get_data_paths()` resolves data across install methods (uv tools, pip, source).

**Data Path Resolution Order**: source/dev mode → uv tool install → pip install → platformdirs → source fallback.

**Package Data** (distributed via `pyproject.toml` shared-data): `.codex/prompts/`, `.opencode/commands/`, `.arckit/templates/`, `scripts/bash/`, `docs/guides/`, plus `docs/{README,DEPENDENCY-MATRIX,WORKFLOW-DIAGRAMS}.md`.

### Slash Command System

Commands exist in multiple formats. The plugin source is canonical; the converter generates the rest.

| AI | Format | Location |
|----|--------|----------|
| Claude Code | `/arckit.requirements` (plugin) | `arckit-claude/commands/requirements.md` |
| Codex CLI | `$arckit-requirements` (skill) | `.agents/skills/arckit-requirements/SKILL.md` |
| Gemini CLI | `/arckit:requirements` (extension) | `arckit-gemini/commands/arckit/requirements.toml` |
| OpenCode CLI | `/arckit.requirements` | `.opencode/commands/arckit.requirements.md` |
| Copilot | `/arckit-requirements` (prompt) | `arckit-copilot/prompts/arckit-requirements.prompt.md` |

**Plugin command frontmatter** supports:

- `description` (required), `effort:` (`low`, `medium`, `high`, `max` for Opus 4.6, `xhigh` for Opus 4.7)
- `keep-coding-instructions: true` (v2.1.94+) — persist instructions across `/compact` for long-running commands
- `handoffs:` — list of `{command, description?, condition?}` entries; converter renders these as a "Suggested Next Steps" section for non-Claude targets

Effort and `keep-coding-instructions` are stripped by the converter for non-Claude targets.

### Agent System

Some commands delegate to autonomous agents (`arckit-claude/agents/arckit-{name}.md`) via the Task tool, for commands that perform extensive web research (>10 WebSearch/WebFetch/MCP calls).

| Agent | Command | Purpose |
|-------|---------|---------|
| `arckit-research` | `/arckit.research` | Market research, vendor evaluation, build vs buy, TCO |
| `arckit-datascout` | `/arckit.datascout` | Data source discovery, API catalogue search |
| `arckit-aws-research` | `/arckit.aws-research` | AWS via AWS Knowledge MCP |
| `arckit-azure-research` | `/arckit.azure-research` | Azure via Microsoft Learn MCP |
| `arckit-gcp-research` | `/arckit.gcp-research` | GCP via Google Developer Knowledge MCP |
| `arckit-framework` | `/arckit.framework` | Transform artifacts into a structured framework |
| `arckit-gov-reuse` | `/arckit.gov-reuse` | Government code reuse via govreposcrape |
| `arckit-gov-code-search` | `/arckit.gov-code-search` | Government code semantic search |
| `arckit-gov-landscape` | `/arckit.gov-landscape` | Government code landscape analysis |
| `arckit-grants` | `/arckit.grants` | UK government grants and funding research |

**Agent frontmatter**: valid fields are `name` (required), `description` (required), `model`, `effort`, `maxTurns`, `tools`, `disallowedTools`, `initialPrompt`. `tools` is an allowlist; `disallowedTools` is a denylist resolved first. Fields `color` and `permissionMode` are invalid in plugin context. Claude-only fields are stripped by the converter. Agents are Claude Code only — other runtimes inline the full agent prompt.

### Three-Tier Reader/Orchestrator/Writer Pattern

Research-heavy agents use a security split to prevent prompt injection from untrusted external content (see `arckit-claude/agents/READER-PATTERN.md`):

| Tier | Touches untrusted bytes? | Holds Write? | Decides scoring? |
|------|--------------------------|--------------|-----------------|
| Reader | **Yes** | No | No |
| Orchestrator | No (validated JSON only) | No | Yes (deterministic, from rubric YAML) |
| Writer | No (structured payload only) | **Yes** | No |

Each tier is a separate subagent with an enforcing `tools` allowlist:

- **Reader**: `WebSearch`, `WebFetch`, `Read`, relevant MCP tools — no `Write`, `Edit`, `Bash`, or `Agent`
- **Orchestrator**: `Agent`, `Bash` (validator + project scripts) — no `WebSearch`, `WebFetch`, untrusted MCP, `Write`, or `Edit`
- **Writer**: `Read`, `Write`, `Edit` only

**Critical constraint**: the orchestrator role must live in the slash command body (`arckit-claude/commands/{name}.md`), **not** in an agent file. Plugin subagents cannot spawn further subagents — only the main thread (slash command body) has access to the `Agent` tool. Reader and writer remain as proper subagent files under `agents/` with `subagent: true` frontmatter (which Claude Code tolerates but doesn't act on; the converter filters it out for non-Claude targets).

Validation between tiers uses `arckit-claude/scripts/validate-handoff.mjs` (zero npm deps) against JSON Schema 2020-12 schemas in `arckit-claude/schemas/`. Scoring rubrics live in `arckit-claude/schemas/scoring-rubrics/`.

### Standalone Commands

`arckit-claude/commands-standalone/` holds commands that declare `allowed-tools:` frontmatter to run in restricted tool environments (e.g. `pages.md` limits itself to Read/Write/Glob). These are plugin-only and not processed by the converter.

### Handoff Schemas

`arckit-claude/schemas/` contains JSON schemas for structured handoff payloads (`datascout-handoff.schema.json`, `gov-reuse-handoff.schema.json`, `grants-handoff.schema.json`) — validated by the corresponding `.mjs` test files in `tests/plugin/`.

### Doc-Types Registry

`arckit-claude/config/doc-types.mjs` is the single source of truth for ArcKit document type codes used by the `validate-arc-filename.mjs` hook and test assertions.

**Triple-registration required when adding/renaming a type code**:
1. `arckit-claude/config/doc-types.mjs` — primary registry
2. `arckit-claude/commands/pages.md` — dashboard allow-list (omitting here silently excludes artifacts from the rendered sidebar)
3. `scripts/bash/generate-document-id.sh` — `MULTI_INSTANCE_TYPES` bash list

### References

`arckit-claude/references/` holds shared prompt-injected reference material:
- `citation-instructions.md` — how commands must add `[DOC_ID-CN]` inline markers and "External References" sections when reading external docs
- `quality-checklist.md` — pre-output checklist injected by commands that produce governance artefacts

### Plugin Skills

Skills live in `arckit-claude/skills/{name}/SKILL.md`. Frontmatter: `name`, `description` (≤1,536 chars), optional `paths:` glob list (v2.1.84+) for auto-activation. The converter strips `paths:` for non-Claude targets.

| Skill | Purpose |
|-------|---------|
| `architecture-workflow` | Onboarding — which command to run and in what order |
| `mermaid-syntax` | Mermaid syntax for `/arckit:diagram` outputs |
| `plantuml-syntax` | PlantUML / C4 syntax |
| `wardley-mapping` | Wardley Map syntax + gameplay/doctrine/climatic patterns |
| `arckit-build` | Bulk-build harness — parallel artefact generation, YAML-recipe-driven, resumable. Claude-only. |

### Plugin User Configuration

Declared in `arckit-claude/.claude-plugin/plugin.json` under `userConfig` (v2.1.83+). Per-field schema: `title`, `type`, `description`, `sensitive: true` (stores in keychain). Substitution uses `${user_config.KEY}`. The converter rewrites to `${KEY}` for non-Claude extensions (shell env var fallback).

Current fields: `GOOGLE_API_KEY`, `DATA_COMMONS_API_KEY`, `organisation_name`, `default_classification`, `governance_framework`.

**MCP `alwaysLoad`** (v2.1.121): `aws-knowledge` and `microsoft-learn` are eagerly loaded; other MCP servers stay deferred.

### Plugin Hooks

Handler files live in `arckit-claude/hooks/`; registered in `arckit-claude/hooks/hooks.json`. Individual entries support an `if:` field (v2.1.85+) using permission rule syntax to narrow triggering. Notable hooks: provenance stamping, manifest auto-update, session telemetry, secret detection, requirement-graph injection. **See [`arckit-claude/hooks/README.md`](arckit-claude/hooks/README.md) for full details.**

### Plugin Monitors

Declared via `monitors` key in `plugin.json` (v2.1.105+); run as persistent subprocesses delivering stdout lines as session notifications. Current monitor: `stale-artifact-scan` — emits alerts for overdue Next Review Dates or stale DRAFTs. Monitors are Claude-only and not propagated to extensions.

### Helper Scripts

`scripts/bash/`:

- `common.sh` — shared utilities sourced by all others (find repo root, slugify, logging)
- `create-project.sh` — creates numbered project dirs, returns JSON
- `generate-document-id.sh` — generates IDs like `ARC-001-REQ-v1.0`. Flags: `--filename`, `--next-num DIR`
- `check-prerequisites.sh` — validates environment
- `list-projects.sh` — lists projects with artifact counts (table or JSON)
- `migrate-filenames.sh` — migrates files to new naming convention (`--dry-run` supported)

Python equivalents of the above live in `scripts/python/` (used by the CLI wheel).

`scripts/converter.py`: config-driven using `AGENT_CONFIG` dict — adding a new AI target requires only a new dict entry. Converts plugin commands to Codex Markdown, Codex Skills, OpenCode Markdown, Gemini TOML, and Copilot prompts. Rewrites `${CLAUDE_PLUGIN_ROOT}` paths, copies supporting files, generates Codex `config.toml`, and rewrites skill command references.

`scripts/managed-agents/arckit-agent.py`: deploys any of the 10 ArcKit agents as Claude Managed Agents via the Anthropic API for headless/CI workflows. Requires `ANTHROPIC_API_KEY`. Run with `--list` to see available agents.

`scripts/bump-version.sh`, `scripts/generate-release-notes.sh`, `scripts/push-extensions.sh`: see [`docs/RELEASING.md`](docs/RELEASING.md) for the full release flow.

## Adding a New Slash Command

1. Create `arckit-claude/commands/{name}.md` with YAML frontmatter (source of truth for Claude Code)
2. Create `.arckit/templates/{name}-template.md` (also copy to `arckit-claude/templates/`)
3. Create `docs/guides/{name}.md` (also copy to `arckit-claude/guides/`)
4. If heavy web research (>10 WebSearch/WebFetch calls), implement the three-tier Reader/Orchestrator/Writer pattern — orchestrator logic goes in the slash command body, reader/writer as `agents/arckit-{name}-reader.md` and `agents/arckit-{name}-writer.md`, schemas in `arckit-claude/schemas/{name}-handoff.schema.json`, tests in `tests/plugin/`
5. Run `python scripts/converter.py` to generate Codex/OpenCode/Gemini/Copilot variants
6. Test plugin in a test repo; test CLI via `arckit init test --ai codex --no-git`
7. Update README.md, docs/index.html, docs/DEPENDENCY-MATRIX.md, CHANGELOG.md

**Command must**: check prerequisites; use `create-project.sh --json` for project path; read its template; **use Write tool** to save output (avoids 32K output token limit); show only a summary; declare `handoffs:` for logical next steps.

### Template Customization

- **Defaults**: `.arckit/templates/` (shipped with ArcKit, refreshed by `arckit init`)
- **User overrides**: `.arckit/templates-custom/` (preserved across updates)

Commands check `templates-custom/` first, fall back to `templates/`. Plugin commands reference templates via `${CLAUDE_PLUGIN_ROOT}/templates/`. When adding/modifying templates, update both `arckit-claude/templates/` and `.arckit/templates/`.

### Document Control Standard

Every template starts with a **Document Control** table followed by **Revision History**.

Standard fields (in order): Document ID, Document Type, Project, Classification, Status, Version, Created Date, Last Modified, Review Cycle, Next Review Date, Owner, Reviewed By, Approved By, Distribution. Domain-specific fields go after.

Valid Classification values: `PUBLIC` / `OFFICIAL` / `OFFICIAL-SENSITIVE` / `SECRET`  
Valid Status values: `DRAFT` / `IN_REVIEW` / `APPROVED` / `PUBLISHED` / `SUPERSEDED` / `ARCHIVED`

**Revision History columns**: Version | Date | Author | Changes | Approved By | Approval Date.

**Standard Footer**:

```markdown
---

**Generated by**: ArcKit `/arckit.{command}` command
**Generated on**: [DATE]
**ArcKit Version**: [VERSION]
**Project**: [PROJECT_NAME]
**Model**: [AI_MODEL]
```

The footer is human-authored. A machine-stamped `## Build Provenance` block is appended automatically by `provenance-stamp.mjs` for artefacts under `projects/**` — templates do not include it.

## Version Management

CLI and plugin versions are independent; all extensions track the plugin. `scripts/bump-version.sh <version>` updates all 15 version-bearing locations in one go. All changes go through a feature branch and are merged to `main` via PR. **Never push directly to `main`.** See [`docs/RELEASING.md`](docs/RELEASING.md).

## Key Patterns

- **Agent Delegation**: Heavy-research commands delegate to agents via the Task tool. The slash command is a thin wrapper. For new research agents, use the three-tier Reader/Orchestrator/Writer pattern.
- **Token Limit Handling**: Commands MUST use Write tool for large documents to avoid the 32K output token limit.
- **Template-Driven Generation**: Never generate freeform documents — always use templates from `.arckit/templates/`.
- **Traceability Chain**: Stakeholders → Goals → Requirements (BR/FR/NFR/INT/DR) → Data Model → Components → User Stories.
- **Citation Traceability**: When commands read external documents, they add inline citation markers (`[DOC_ID-CN]`) and an "External References" section. See `arckit-claude/references/citation-instructions.md`.
- **Requirement ID Prefixes** (numeric suffix 1–3 digits): `BR-xxx` Business, `FR-xxx` Functional, `NFR-xxx` Non-Functional (NFR-P/NFR-SEC/etc.), `INT-xxx` Integration, `DR-xxx` Data
- **UK Government Context**: Many commands target UK public sector (GDS Service Standard, TCoP, NCSC CAF, Orange/Green Book, G-Cloud/DOS procurement). See README.md for full compliance coverage.
- **Wardley Map Format**: OnlineWardleyMaps syntax for visualization at <https://create.wardleymaps.ai>.
